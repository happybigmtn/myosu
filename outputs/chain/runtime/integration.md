## Where This Runtime Fits

The `myosu-runtime` is the execution layer for the myosu chain. It sits in the workspace hierarchy as:

```
Cargo workspace
└── crates/myosu-chain/          # Workspace member (lib = "src/lib.rs")
    ├── runtime/                 # myosu-runtime (this build)
    │   ├── Cargo.toml
    │   ├── build.rs            # WASM builder
    │   └── src/
    │       ├── lib.rs          # Runtime definition
    │       └── chain_spec.rs   # Development genesis
    ├── node/                   # Phase 2: myosu-node
    ├── common/                 # Phase 2: myosu-chain-common
    └── pallets/
        └── game-solver/        # Stage 0 pallet (already buildable)
```

## How the WASM Blob Is Used

The `substrate-wasm-builder` produces two WASM files:

1. **`myosu_runtime.wasm`** — The raw runtime WASM blob, loaded by the Substrate node at chain initialization
2. **`myosu_runtime.compact.wasm`** / **`.compressed.wasm`** — Size-optimized variants

The node (Phase 2) will load this WASM via `sc_executor::WasmExecutor` and use it to validate extrinsics. The native lib (`libmyosu_runtime.rlib`) is used for host-side validation; the WASM is used for cross-runtime (validator) validation.

## Runtime API Surface

The runtime exposes these APIs (via `impl_runtime_apis!`):

| API | Version | Purpose |
|-----|---------|---------|
| `Core` | 1 | Runtime version, `execute_block`, `initialize_block` |
| `Metadata` | 1 | `metadata()`, `metadata_at_version`, `metadata_versions` |
| `BlockBuilder` | 1 | `apply_extrinsic`, `finalize_block`, `inherent_extrinsics`, `check_inherents` |
| `TaggedTransactionQueue` | 1 | `validate_transaction` |
| `OffchainWorkerApi` | 1 | `offchain_worker` |
| `SessionKeys` | 1 | `generate_session_keys`, `decode_session_keys` |

Phase 2 node's RPC and consensus will consume these APIs.

## Genesis Integration

The runtime's `RuntimeGenesisConfig` (auto-generated by `construct_runtime!`) structures genesis for:

- `System`: `code` (WASM blob), `消耗` ( Trie ID, etc.)
- `Balances`: `balances: Vec<(AccountId, Balance)>`
- `Sudo`: `key: Option<AccountId>`
- `Timestamp`: `minimum_period: u64`

Phase 2's `myosu-node` will extend this via `chain_spec.rs` with endowed accounts, initial balances, and sudo key configuration.

## Integration Points With `myosu-games`

The `myosu-games` crate (upstream) uses `pallet-game-solver` which lives under `crates/myosu-chain/pallets/game-solver/`. The game-solver pallet is frame-based, so it needs a `frame_system` and `frame_support` dependency.

When Phase 2 adds `myosu-node` with `pallet-aura` (or manual seal), the game-solver pallet can be added to the runtime's `construct_runtime!` block:

```rust
construct_runtime! {
    pub enum Runtime {
        System: frame_system,
        Balances: pallet_balances,
        Sudo: pallet_sudo,
        Timestamp: pallet_timestamp,
        GameSolver: pallet_game_solver,  // Phase 2 or 3
    }
}
```

This is the natural integration point — no changes to the runtime API surface needed; the pallet is added to `construct_runtime!` and its dispatchable calls become part of `RuntimeCall`.

## Build System Notes

### `CARGO_TARGET_DIR` requirement

The `substrate-wasm-builder` runs a nested `cargo metadata` during the WASM build phase. If `CARGO_TARGET_DIR` is set to a path with stale build artifacts from a different worktree, the wasm-builder will fail with:

```
Failed to find entry for package myosu-runtime ("/path/to/Cargo.toml")
```

**Fix**: Always unset or override `CARGO_TARGET_DIR` when building:
```bash
CARGO_TARGET_DIR=/path/to/worktree/target cargo build -p myosu-runtime --release
```

This is an environment issue, not a code issue.
