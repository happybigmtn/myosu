# Phase 1: THINK (Deep Codebase Review)

You must review the actual source code first, not just docs and plans.

## Mandatory review lenses

Review the codebase through these lenses and ground every claim in specific files:

1. Engineering review
- Read the main entry points, core modules, state definitions, and binary entry points.
- Map architecture, data flow, failure modes, observability gaps, deployment risks, and coupling.
- Flag unwraps, TODO/FIXME, dead code, unbounded collections, and weak rollback posture.

2. Security and trust-boundary review
- Trace auth, input validation, secrets handling, injection risks, shell execution, privilege boundaries, and replay/authorization issues.
- For each issue, record threat, likelihood, impact, and current mitigation.

3. Design and UX review
- Review all user-facing routes and the key hooks/components that drive them.
- Rate information architecture, interaction-state coverage, responsive intent, accessibility, and AI-slop risk.
- If a design system exists, check drift against it.

4. Test and CI review
- Read CI config and the main test suites.
- Build a coverage map per major feature: unit, integration, e2e, and edge-case coverage.
- Flag fake-green tests, over-mocked tests, disabled suites, and weak failure-path coverage.

5. Ops and strategic review
- Read git history, doctrine files, plans, specs, existing genesis docs, and deployment/ops scripts.
- Build an implementation-status table for prior claims: verified, partially verified, stub only, or not found.
- Identify hotspots, velocity, contributor patterns, and stale documentation references.

## Cross-reference rules

- Read all top-level doctrine files that exist: `README.md`, `GOAL.md`, `AGENTS.md`, `SPEC.md`, `INVARIANTS.md`, `DESIGN.md`, deployment docs, and similar repo doctrine.
- Read every file under `plans/`, `specs/`, and any existing `genesis/`.
- Treat completed `[x]` milestones as claims that must be verified against code.
- Never trust docs over code. If docs claim something that the code does not support, say so clearly.

## Git and evidence review

Extract repo evidence:
- `git log --oneline -100`
- `git shortlog -sn --since="3 months ago"`
- hotspot files from recent history
- fix ratio, test-touch ratio, contributor breakdown, and shipping velocity
- proof artifacts, deployment evidence, and stale-doc references if present

## ASSESSMENT.md requirements

Write `genesis/ASSESSMENT.md` with:
- the reframe: what the project says it is vs what the code shows it actually is
- one-sentence project summary
- six forcing questions: demand, status quo, desperate specificity, narrowest wedge, surprise, future-fit
- source-code findings by component with specific file paths
- what works, what is broken, what is half-built
- tech-debt inventory
- security risks
- test gaps
- git metrics summary
- documentation staleness table
- implementation-status table for prior claims and plans
- code-review coverage list proving which source files were actually read

# Phase 2: PLAN (Full Review Stack)

Use the assessment to shape a high-quality planning corpus:

- CEO lens: challenge product framing, identify the narrowest wedge, and prefer reversible plans.
- Engineering lens: map milestones to concrete files, architecture, failure modes, observability, and deployment safety.
- Design lens: specify states, responsiveness, accessibility, and avoid generic SaaS-template thinking.
- Do not restart already-proven work. Mark pre-satisfied work explicitly where the code review shows it already exists.

# Phase 3: BUILD (Write the Corpus)

Write these files under `genesis/`:

1. `ASSESSMENT.md`
2. `SPEC.md`
3. `PLANS.md`
4. `GENESIS-REPORT.md`
5. `DESIGN.md` if the repo has meaningful user-facing surfaces or needs a fresh design system
6. `plans/001-master-plan.md`
7. `plans/002-*.md` through `plans/NNN-*.md`

`SPEC.md` should explain what the project is, who it is for, the current architecture, tech stack, and major decisions already made.

`PLANS.md` should document the ExecPlan conventions used in the numbered plans.

`plans/001-master-plan.md` should be the 180-day roadmap that orders the numbered workstreams.

Every numbered plan must follow this exact skeleton:

```md
# NNN - <Short, action-oriented title>

## Purpose / Big Picture

## Context and Orientation

## Architecture

## Progress
- [x] (pre-satisfied) M1. ...
- [ ] M2. ...
  - Surfaces: `path/a`, `path/b`
  - What exists after: ...
  - Why now: ...
  - Proof: `specific command`
  - Tests: `specific test command`

## Surprises & Discoveries

## Decision Log
- Decision: ...
  - Why: ...
  - Failure mode: ...
  - Mitigation: ...
  - Reversible: yes/no

## Validation and Acceptance

## Outcomes & Retrospective
_Updated after milestones complete._
```

Plan quality requirements:
- 3-8 milestones per plan
- 80-200 lines per plan unless there is a compelling reason otherwise
- concrete repo-relative file paths in Surfaces
- specific proof commands, not vague ones like `cargo test`
- include any tech debt, docs repair, testing gaps, security work, and missing infrastructure exposed by review
- do not silently drop existing plans; carry them forward, rewrite them, split them, or explicitly explain why they are dropped

# Phase 4: REVIEW

Before finishing, self-review the corpus:
- every plan has enough context for a new contributor
- every milestone names specific files
- every plan includes at least one failure scenario
- every claimed completed capability matches code reality
- no plan re-implements something the review verified as already working
- the master plan references the numbered plans

If a plan is too thin, vague, or redundant, fix it before finishing.
