Canonical spec contract:
# Spec Authoring Canon

Status: Active
Date: 2026-04-01
Governs: `specs/*.md`

This document is the canonical delivery shape for LLM-generated specs in
Malina. It applies both when creating new specs from planning material and when
reverse-engineering specs from an existing codebase.

The purpose of a spec is to define `WHAT` must be true, not `HOW` to implement
it. Specs are behavioral contracts. `IMPLEMENTATION_PLAN.md` is where required
tests, sequencing, and implementation tasks belong.

If a generated spec does not follow this document, the spec is wrong.

## Core Principles

- Specs describe behavioral outcomes, observable results, and scope boundaries.
- Specs do not prescribe implementation approach, file-by-file tasking, or
  code structure.
- Specs must be self-contained enough that a build agent can understand the
  capability without chat history.
- Specs must be narrow enough to pass the topic scope test.
- Reverse-engineered specs document reality, not intent. If code is buggy, the
  spec describes the buggy behavior until a new desired-behavior spec replaces
  it.

## Hard Rules

- Never include code blocks.
- Never suggest variable names, class names, or specific implementation
  techniques.
- Never put required tests, proof commands, rollout steps, or task sequencing in
  the spec. Those belong in `IMPLEMENTATION_PLAN.md`.
- Never merge unrelated capabilities into one spec just because the same files
  change.
- Never describe a topic in a way that needs the word `and` to glue together
  multiple independent capabilities.

## Topic Scope Test

Every spec must pass the "one sentence without `and`" test.

Good:

- "The queue compiler turns a reviewed plan into admitted executable tasks."
- "The house agent manages automated play sessions with explicit guardrails."

Bad:

- "The queue system parses plans and runs reviews and updates live status."
- "The user system handles authentication, profiles, and billing."

If the capability needs `and` to describe it, split the spec.

## What Belongs In A Spec

- The topic statement: one coherent capability
- Why the capability matters at the system level
- Scope boundaries
- Current state or existing reality
- Existing leverage or reused flows
- Non-goals
- Behavioral description of what the system does
- Acceptance criteria written as observable outcomes

## What Does Not Belong In A Spec

- Code snippets
- Commands to run
- Required tests
- Proof commands
- Task packets
- File ownership lists for implementation
- Step-by-step implementation instructions
- Architecture decisions that dictate the solution unless they are already part
  of the required behavior or current system reality

## Canonical Spec Shape

Every generated spec should use this shape unless there is a very strong reason
not to.

- Title line: `# Specification: {Title}`
- Header fields:
  `Source: {What motivated this spec or what reality it was reverse-engineered from}`
  `Status: Draft`
  `Depends-on: {Other spec topics or "none"}`
- Sections, in order:
  `## Purpose`
  `## Whole-System Goal`
  `## Scope`
  `## Current State`
  `## What Already Exists`
  `## Non-goals`
  `## Behaviors`
  `## Acceptance Criteria`

Expected content by section:

- `## Purpose`
  3-8 sentences describing the user, operator, runtime, or downstream outcome.
  Lead with the outcome, not the mechanism.
- `## Whole-System Goal`
  `Current state:`
  `This spec adds:`
  `If all ACs land:`
  `Still not solved here:`
- `## Scope`
  `In scope:`
  `Out of scope:`
- `## Current State`
  Current truth surfaces, current behavior, or an explicit statement that the
  capability does not yet exist.
- `## What Already Exists`
  A short table with columns:
  `Sub-problem | Existing code / flow | Reuse / extend / replace | Why`
- `## Non-goals`
  Specific adjacent expectations this spec intentionally does not cover.
- `## Behaviors`
  Behavior in execution order when helpful, including notable state transitions
  and important failure-path behavior.
- `## Acceptance Criteria`
  Observable outcomes only.

## Section Guidance

### Header

- `Source` is required.
- `Status` should normally be `Draft` when first generated.
- `Depends-on` should reference other spec topics, not task IDs.

### Purpose

- State why this capability matters.
- Name the primary consumer of the outcome when relevant: user, operator,
  runtime, or downstream system.
- Do not drift into implementation plan territory.

### Whole-System Goal

- This section keeps the spec tied to system value rather than local edits.
- If this section reads like a tiny refactor with no capability consequence, the
  spec is probably too narrow or should be part of another spec.

### Scope

- Be explicit about boundaries.
- Out-of-scope items should be specific and plausible, not generic filler.

### Current State

- For greenfield work, say that the capability does not yet exist.
- For brownfield work, summarize the current behavior or truth surfaces.
- File paths are allowed here when they describe existing reality.

### What Already Exists

- This section prevents accidental reinvention.
- At least one row should mention a real file path, subsystem, or runtime flow
  when such leverage exists.
- If nothing exists yet, say so explicitly.

### Non-goals

- This section is mandatory.
- Use it to stop scope creep and to protect neighboring systems from accidental
  widening.

### Behaviors

- Focus on what happens, in what order, and what outcomes are visible.
- Reverse-engineered specs should document actual runtime behavior, including
  surprising behavior and failure behavior.
- Do not prescribe technical implementation.

### Acceptance Criteria

- Acceptance criteria must be behavioral outcomes only.
- They must be observable and verifiable, but they should not specify the test
  command.
- Good ACs describe what success looks like from the outside.

Good:

- "The queue compiler produces blocked tasks when prerequisites are incomplete."
- "The house agent stops automated play when the configured loss limit is hit."
- "The operator can see which spec is waiting on review."

Bad:

- "Use a Rust enum for task state."
- "Add a `compile_queue()` function."
- "Run `cargo test -p malina-plan`."

## Reverse-Engineering Rules

When deriving a spec from code:

- The code is the source of truth.
- Document actual behavior, not desired behavior.
- Trace entry points, branches, state transitions, side effects, and error
  handling before writing the spec.
- Strip implementation references from the final prose unless they are needed in
  `Current State` or `What Already Exists`.
- If current code and current docs disagree, the spec follows the code.

## Duplication Rules

- One capability per spec.
- Shared behavior may be referenced across specs, but duplicated full specs are
  not allowed.
- Update an existing spec instead of creating a near-duplicate.
