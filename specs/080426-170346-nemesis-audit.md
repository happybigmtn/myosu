## Finding NEM-001: Validator score is self-referential in E2E happy-path

### Affected surfaces
- `crates/myosu-validator/src/validation.rs:217-260` — `score_response()` loads checkpoint and computes expectation from same source
- `tests/e2e/local_loop.sh:267-290` — validator scoring with `--checkpoint` from miner-produced file
- `tests/e2e/validator_determinism.sh` — feeds miner checkpoint back into validator

### Triggering scenario
1. Miner runs with `--train-iterations 0` and produces checkpoint at `latest.bin`
2. Miner also produces `--response-file` from its solver answering the query
3. Validator loads the same checkpoint via `--checkpoint "$checkpoint_path"`
4. Validator computes `solver.answer(query)` — identical policy that produced miner's response
5. L1 distance = 0.0, score = 1.0, `exact_match=true`

**Result**: The happy-path E2E reports perfect score on a zero-iteration checkpoint. This is a self-consistency proof, not a quality proof.

### Invariant or assumption that breaks
- INV-003 (Game Verification Determinism): The proof tests determinism correctly, but the happy path does not test meaningful quality discrimination.
- The AGENTS.md operator loop documents this as intentional but does not flag the limitation explicitly in the proof surfaces.

### Why this matters now
A validator submitting weights for a zero-iteration miner receives the same high weights as a 512-iteration miner on the current E2E harness. The Yuma Consensus reward distribution is not tested against a genuine quality gradient in the default proof path.

### Discovery path
**State** — traced `score_response()` code path, confirmed it loads checkpoint and computes `solver.answer()` from that checkpoint. Cross-referenced `local_loop.sh` to confirm the same checkpoint is fed back into the validator.

## Finding NEM-002: Stage0NoopSwap has unbounded max_price — slippage time-bomb

### Affected surfaces
- `crates/myosu-chain/runtime/src/lib.rs:94-97` — documented caveat: "max_price() intentionally returns the currency max value"
- `crates/myosu-chain/runtime/src/lib.rs:132-137` — `Stage0NoopSwap::max_price<C: Currency>() -> C { C::MAX }`
- `crates/myosu-chain/pallets/game-solver/src/staking/add_stake.rs:78` — `T::SwapInterface::stage0_max_price()` used as price limit
- `crates/myosu-chain/pallets/game-solver/src/coinbase/run_coinbase.rs:141` — `swap_tao_for_alpha(..., stage0_max_price(), ...)` in `inject_and_maybe_swap`

### Triggering scenario
1. Stage-0: swaps are identity (1:1, no pool state), so unbounded price limit causes no damage
2. Future runtime replaces `Stage0NoopSwap` with a real AMM but forgets to replace the price limit
3. Staking and emission code paths still call `stage0_max_price()` — the new AMM receives unbounded order
4. Attacker exploits unbounded slippage tolerance on large stake/swap operations

### Invariant or assumption that breaks
- INV-005 (Plan And Land Coherence): documented caveat exists but there is **no compile-time assertion, no migration guard, and no try_state check** that forces a real price limit before AMM substitution.

### Why this matters now
No active damage in stage-0. This is a **hardened time-bomb**: it will compile silently with a real AMM and expose real token amounts to slippage exploitation. Risk grows as system approaches token-economics spec work.

### Discovery path
**Feynman** — traced `add_stake` flow through swap calls to `stage0_swap_tao_for_alpha`, noted `stage0_max_price()` returns `C::MAX`, compared against runtime comment acknowledging risk without enforcement.

## Finding NEM-003: Epoch per-UID emission accumulation has untested truncation gap

### Affected surfaces
- `crates/myosu-chain/pallets/game-solver/src/epoch/run_epoch.rs:490-530` (both `epoch_mechanism` and `epoch_dense_mechanism`)
- `crates/myosu-chain/pallets/game-solver/src/coinbase/run_coinbase.rs:47-65` — `close_integer_emission_split` 
- `crates/myosu-chain/pallets/game-solver/src/tests/stage_0_flow.rs:599-633` — existing sweep

### Triggering scenario
The `stage_0_coinbase_truncation_dust_is_closed_exactly_sweep` test verifies the coinbase block-emission → owner/server/validator split closes exactly. This is solid.

**What the sweep does NOT cover**: the per-UID epoch emission values computed in `run_epoch.rs`:

```rust
let server_emission: Vec<AlphaCurrency> = normalized_server_emission
    .iter()
    .map(|se: &I32F32| I96F32::saturating_from_num(*se).saturating_mul(float_rao_emission))
    .collect();
let server_emission: Vec<AlphaCurrency> = server_emission
    .iter()
    .map(|e: &I96F32| e.saturating_to_num::<u64>().into())  // <-- per-UID floor
    .collect();
```

Each UID's emission is independently floored from `I96F32` to `u64`. The sum of per-UID emissions may be less than the total epoch allocation by up to `n_validators + n_miners` rao per epoch.

### Invariant or assumption that breaks
- The emission accounting gate: "sum(distributions) == block_emission * epochs" 
- Per-UID epoch truncation could cause silent leakage of up to N rao per epoch (where N = number of UIDs).

### Why this matters now
The coinbase split is proven closed, but the epoch distribution accumulation path is not. At scale, this drift could accumulate. The stage-0 emission sweep does not cover this path.

### Discovery path
**Feynman** — traced `stage_0_coinbase_truncation_dust_is_closed_exactly_sweep` scope, cross-referenced `close_integer_emission_split` against the epoch per-UID emission computation in `run_epoch.rs`. Identified that per-UID truncation floors are not bounded in tests.

## Finding NEM-004: No HTTP axon integration test for miner-validator scoring

### Affected surfaces
- `crates/myosu-miner/src/axon.rs` — HTTP axon server
- `crates/myosu-validator/src/chain.rs` — miner discovery from chain
- `crates/myosu-chain/pallets/game-solver/src/serving.rs` — `serve_axon` extrinsic
- `tests/e2e/local_loop.sh` — uses file-based query/response, not HTTP for scoring

### Triggering scenario
1. Miner binary calls `serve_axon` and starts HTTP server
2. Validator binary queries `all_axons` RPC, gets miner's IP/port
3. Validator makes HTTP POST to miner `/strategy` with wire-encoded `StrategyQuery`
4. Miner responds with `StrategyResponse`
5. Validator scores the response and submits weights

**This end-to-end flow has no automated test.** The `local_loop.sh` uses file-based query/response (`--query-file`, `--response-file`) which bypasses HTTP entirely for the scoring validation. HTTP axon is only tested for the gameplay smoke test at the end.

### Invariant or assumption that breaks
- INV-005 (end-to-end wiring): The HTTP wire protocol between validator and miner is a critical trust boundary. Any codec mismatch, HTTP header issue, or path error would not be caught by the file-based E2E harness.

### Why this matters now
The protocol is defined by `myosu-games-poker/src/wire.rs` but has no integration test coverage through the HTTP layer. A regression in the axon HTTP handler or wire codec would not be detected by CI.

### Discovery path
**State** — cataloged miner-validator HTTP interaction paths, compared against `local_loop.sh`, confirmed no HTTP integration test exists for the scoring path.

## Finding NEM-005: INV-004 enforcement is textual and misses transitive deps

### Affected surfaces
- `crates/myosu-play/tests/invariants.rs:35-48` — `inv_004_solver_and_gameplay_bins_do_not_depend_on_each_other`
- `INVARIANTS.md:INV-004` — "no direct imports" policy

### Triggering scenario
The current test runs `cargo tree -p myosu-play` and asserts the output does not contain "myosu-miner". This only prevents **direct** `myosu-play → myosu-miner` edges.

A transitive import through a shared dependency (e.g., `myosu-play → myosu-chain-client → myosu-miner` if chain-client re-exports miner types) would **not** be caught. The enforcement is pattern-matching on cargo tree output, not a structural ownership boundary.

### Invariant or assumption that breaks
- INV-004: "no direct imports" is enforced but transitive imports through shared deps are not caught.
- Current boundary is clean (`cargo tree -p myosu-play` confirms no miner), but enforcement is brittle.

### Why this matters now
If the workspace structure changes (e.g., `myosu-chain-client` re-exports miner-specific types), the textual check could pass while the semantic boundary is violated.

### Discovery path
**State** — reviewed test implementation, traced enforcement mechanism to textual `cargo tree` pattern-matching, identified transitive-import gap.

## Finding NEM-006: try_state emission guard has poor diagnostics

### Affected surfaces
- `crates/myosu-chain/pallets/game-solver/src/utils/try_state.rs:20-30` — `check_total_issuance()`
- `crates/myosu-chain/pallets/game-solver/src/lib.rs:75` — `TOTAL_ISSUANCE_TRY_STATE_ALERT_DELTA = 1`

### Triggering scenario
The `try_state` hook computes `expected_total_issuance` and compares against `TotalIssuance::<T>::get()`. If `diff > 1` rao, it returns an error with message:

```rust
ensure!(diff <= delta, "TotalIssuance diff greater than allowable delta")
```

This message does not include:
- The actual diff magnitude
- The expected vs live values
- Which component caused the drift

If the guard triggers on mainnet, the node would panic on `on_initialize` with a generic message, making root-cause analysis difficult.

### Invariant or assumption that breaks
- INV-005 diagnostic quality: The 1-rao guard is the emission accounting no-ship gate, but poor diagnostics impede incident response.

### Why this matters now
The `try_state` runs on every block in test/try-runtime builds. If emission accounting drifts, the diagnostic message is insufficient for rapid debugging.

### Discovery path
**State** — read `utils/try_state.rs`, confirmed assertion lacks magnitude disclosure in error message.

2. `check_robopoker_fork_status.sh` runs, compares workspace pin against upstream, reports divergence counts
