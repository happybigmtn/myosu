# Upgrade Research Solvers from Metadata Heuristics to Typed State-Aware Engines

This ExecPlan is a living document. The sections `Progress`, `Surprises & Discoveries`, `Decision Log`, and `Outcomes & Retrospective` must be kept up to date as work proceeds.

`PLANS.md` is checked into the repository root and this document must be maintained in accordance with it. This plan assumes the existing research portfolio surfaces are already landed: `crates/myosu-games-portfolio` answers every portfolio-routed research game, `tests/e2e/research_strength_harness.sh` proves the typed strength roundtrip, and `tests/e2e/research_play_harness.sh` proves the bounded core/playtrace surface across the full corpus.

## Purpose / Big Picture

After this plan is implemented, the portfolio solvers will stop pretending that one bootstrap label is enough state for twenty different games. Every portfolio-routed game will answer from typed challenge data that exposes the actual strategic pressure of the representative spot: stack and equity pressure for poker-like games, contract and trump pressure for trick-taking games, safety and ukeire for mahjong, row-strength and foul margin for OFC, and so on.

The visible result is that `strength_query`, `strength_roundtrip`, miner responses, validator reports, and play smoke output still use the same operator-facing commands, but the returned strategy is now driven by game-specific state instead of by `challenge_id` string matching and one fixed canned branch per game. A reader should be able to inspect `crates/myosu-games-portfolio/src/state.rs`, see typed per-family challenge payloads, and then trace those fields directly into the scoring logic of the matching engine family.

## Context and Orientation

The work lives in `crates/myosu-games-portfolio`. That crate is the compact research-game adapter used by the wider Myosu operator surfaces. It does not claim to solve every game in `research/game-rules`; instead it gives each game a deterministic solver checkpoint, query format, response format, and first strength benchmark so miners, validators, and the play smoke surface can all route the same game identity.

The important files are:

- `crates/myosu-games-portfolio/src/state.rs`, which currently defines `PortfolioChallenge` and `PortfolioChallengeSpot`. This is the typed challenge schema for the portfolio strength route.
- `crates/myosu-games-portfolio/src/protocol.rs`, which wraps `PortfolioChallenge` inside `PortfolioStrengthInfo` and `PortfolioStrengthQuery`.
- `crates/myosu-games-portfolio/src/engines/mod.rs`, which dispatches a typed challenge to the appropriate family engine.
- `crates/myosu-games-portfolio/src/engines/*.rs`, which implement the current family heuristics.
- `crates/myosu-games-portfolio/src/solver.rs`, which exposes `PortfolioSolver::strength_query`, `answer_strength_checked`, and `strength_quality`.
- `crates/myosu-games-portfolio/examples/strength_query.rs`, `strength_roundtrip.rs`, and `strength_manifest.rs`, which are the operator-visible examples and proof helpers.
- `tests/e2e/research_strength_harness.sh`, `tests/e2e/research_portfolio_harness.sh`, `tests/e2e/research_games_harness.sh`, and `tests/e2e/research_play_harness.sh`, which are the all-corpus proof surfaces that must stay green.

A typed challenge is the serializable description of one representative decision spot for a game. It is not the whole game tree and it is not just a display label. It must carry enough state that the matching engine can decide between its legal high-level actions without inventing hidden assumptions off to the side.

## Plan of Work

First, replace the current one-struct-fits-all challenge schema in `crates/myosu-games-portfolio/src/state.rs` with typed family payloads. Each payload keeps a `PortfolioChallengeSpot` field so the challenge still carries `challenge_id`, `decision`, `rule_file`, and `solver_family`, but it also adds the small game-state facts that matter to the engine family. The poker-like family needs stack, pot, hand-strength, draw-strength, and pressure fields. The trick-taking family needs trump, winners, contract pressure, and penalty pressure. The shedding family needs control combinations, bomb pressure, and opponent card counts. The descriptive spot should still contain the stable game id in `challenge_id`, but it should name the scenario in plain English, for example a bridge contract-play spot or a backgammon bear-off race spot. The new bootstrap state remains one spot per game for now; the goal of this pass is stronger state-aware reasoning, not a multi-scenario catalog.

Third, change `crates/myosu-games-portfolio/src/engines/mod.rs` and each family engine file to consume the typed challenge payload directly. The family module should not manufacture its own hidden reference state once the challenge already carries the same information. The scoring logic should stay compact and deterministic, but the weights should now be functions of the typed fields. That keeps the engine honest: if the query says the hand is strong and in position, the poker-like engine should lean toward value or pressure; if the query says the player faces a riichi threat with thin ukeire, the mahjong engine should lean toward folding.
